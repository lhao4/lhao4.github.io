底层学习

从零开始：现代编译器优化、性能优化与二进制优化的全面详解

第一部分：硬件瓶颈与操作系统底层原理——为什么优化必不可少 现代优化源于CPU与内存的“速度鸿沟”（内存延迟数百周期）。操作系统（Linux/Android/iOS）管理虚拟内存、二进制加载，进一步放大瓶颈。 1. CPU体系结构基础与前端瓶颈 流水线（Pipeline）：CPU分取指（Fetch）、解码（Decode）、执行（Execute）、访存（Memory）、写回（Writeback）。现代CPU超长流水线（Intel 14-19级，Arm 8-15级）。 乱序执行（Out-of-Order Execution）：CPU重排指令隐藏延迟，但前端（取指/解码）仍是瓶颈。 指令缓存（L1i Cache）：x86/Intel典型32-64KB/核，Arm AArch64类似32KB。过去20年容量几乎不变（Moore定律偏向核数而非缓存）。 缓存缺失（Instruction Cache Miss）：代码不局部，CPU等主存（数百周期罚时）。 例子：大型二进制（>100MB）热代码散布，L1i命中率<90%，IPC降30%。 分支预测（Branch Prediction）： 动态分支（if/loop）占20-30%。预测器（TAGE/SC等）猜错罚10-20周期。 x86 CISC指令长度可变（1-15字节），预测更难；Arm RISC固定4字节，更友好。 TLB（Translation Lookaside Buffer）： 虚拟到物理地址翻译缓冲。iTLB典型48-64条目。 代码跨多4KB页，TLB Miss慢（页表行走）。 Huge Pages（2MB/1GB）：减TLB压力，BOLT/Propeller常选性用。 指令预取（Instruction Prefetcher）：硬件提前取下一页，布局乱失效。 2. 操作系统视角：二进制加载与执行 ELF（Linux/AArch64）、Mach-O（iOS）、PE（Windows）格式： 段（.text代码、.data数据）、符号表、重定位。 加载时OS映射.text到虚拟内存，按需分页（Demand Paging）。 缺页故障（Page Fault）： 移动端关键：App启动冷代码未载入，主存/闪存取慢（ms级）。 Android/iOS Art/Dalvik进一步JIT/AOT放大。 虚拟内存与ASLR（Address Space Layout Randomization）： 安全随机化布局，破坏静态优化。BOLT等后链工具重排后保持兼容。 x86 vs ARM/AArch64差异： x86（CISC）：指令密，分支多，缓存压力大；iTLB小。 ARM/AArch64（RISC）：固定长指令，条件执行减分支；Thumb2压缩模式移动端用。 AArch64服务器（如AWS Graviton）前端瓶颈类似x86，但分支预测更强（Arm Neoverse）。 优化目标：提高IPC（目标4-6），减前端摊销（Front-End Stalls）。 第二部分：编译原理底层——从源码到二进制全流程 编译器（如GCC/Clang/LLVM）分三阶段： 前端（Front-End）： 词法/语法分析 → AST → 类型检查。 生成IR（Intermediate Representation，如LLVM IR，SSA形式）。 中端（Middle-End）： IR优化：常量折叠、传播、死代码消除、内联、循环向量化。 Pass Manager：LLVM模块化Pass链。 后端（Back-End）： 指令选择、寄存器分配、调度、代码生成。 输出汇编 → 汇编器 → 对象文件。 链接器（ld/lld）合并对象 → 可执行二进制。LTO在链接时回IR优化。 ...
